RAG that actually works at scale

A basic RAG (one dense embedding + top-k) breaks down on a real corpus. Here is how we upgraded it into a hybrid, re-ranked pipeline over 5,000 VWO test cases — and what every piece does.

bge-m3 (dense + sparse) Qdrant RRF fusion bge-reranker-v2-m3 query rewriting Groq gpt-oss-120b

Why basic RAG isn't enough

A single dense vector captures meaning but misses exact tokens. On a QA corpus you constantly search for things dense embeddings blur together.

The problems

→ A query for VWO-3400 returns semantically similar tickets, not the exact one.
→ "security test cases for the SDK" retrieves generic security text, missing the module keyword.
→ Top-k by cosine similarity is a coarse ranking — the best chunk is often at rank 7, not rank 1.

The fixes (this app)

→ Sparse vectors catch exact IDs and keywords.
→ RRF fuses semantic + keyword rankings.
→ A cross-encoder re-reads query+chunk together to find the truly best few.
→ Query rewriting widens recall before any of that.

The pipeline, end to end

Two phases. Ingestion builds the hybrid index once; every chat query flows through retrieval, fusion, reranking, and generation.

The four techniques

Each one fixes a specific failure of naive retrieval.

densesparse Hybrid embeddings

One model, bge-m3, emits both a 1024-dim dense vector (meaning) and a sparse vector (token weights). Dense finds paraphrases; sparse nails exact IDs and keywords.

dense: [-0.040, -0.019, -0.041, -0.044, -0.007, ...]
sparse: Code 0.32collection 0.27148 0.26Smart 0.24

RRF Reciprocal Rank Fusion

Dense and sparse scores live on different scales — you can't just add them. RRF merges by rank: each list contributes 1 / (k + rank). Robust, tuning-free.

score(doc) = 1/(60+rdense) + 1/(60+rsparse)

A chunk ranked #6 by dense and #2 by sparse can beat one that was #1 in only a single list.

rerank Cross-encoder

bge-reranker-v2-m3 reads the query and each candidate together and outputs a relevance score — far sharper than comparing two independent vectors. We rerank the fused pool down to the final 4.

Only the top few (tall bars) reach the LLM.

rewrite Query rewriting

Before retrieval, the LLM expands the question into 3 alternate phrasings. More phrasings → more surface area for both dense and sparse search → higher recall.

"security tests for SDK" "auth/injection SDK cases" "server-side API security"

What happens on one query

RewriteThe question becomes 3 alternate phrasings (Groq).
Embed & searchEach phrasing is embedded with bge-m3; dense and sparse each return their top-20 from Qdrant.
Fuse (RRF)The two ranked lists merge into one fused candidate pool.
Re-rankbge-reranker-v2-m3 re-scores the pool; the top-4 chunks survive.
GenerateGroq gpt-oss-120b answers using only those 4 chunks, citing each as [Chunk N].
Two auto-detected modes: Answer (grounded Q&A) and Generate (e.g. "create a test case for VWO-3400 heatmap privacy masking" — produces a structured test case using the retrieved cases as templates).
