What is QABuddy.ai?

QABuddy.ai is an AI-powered assistant for QA engineers that provides cited answers grounded in your team's actual QA knowledge base — Selenium framework code, Playwright tests, test cases, JIRA tickets, PRDs, Jenkins logs, meeting transcripts, and glossaries.

Unlike generic AI chatbots that hallucinate, QABuddy retrieves the most relevant chunks from your indexed knowledge base using BM25 search, then passes them as grounded context to the LLM. Every answer includes numbered citations pointing back to the exact source.

Key capabilities:

  • Answer mode — practical QA engineering answers with code examples
  • Generate mode — auto-detected when you ask to create/write test cases
  • Review mode — coverage gap analysis against requirements
  • RCA mode — root cause analysis for test failures and flakiness

Architecture Overview

The system consists of two main pipelines:

Ingestion Pipeline (build-time)

  1. Parse — load raw files (PDF, CSV, JSON, Java, TypeScript, Markdown)
  2. Clean — normalize whitespace, remove noise, deduplicate
  3. Chunk — split into ~512 token chunks with 64 token overlap
  4. Embed — generate dense (semantic) + sparse (BM25) vectors via BGE-M3
  5. Index — upsert to Qdrant vector DB with metadata (source_type, path, ticket_key, etc.)

In the Vercel-optimized version, the knowledge base is pre-compiled as a static TypeScript module (lib/knowledge-base.ts), and BM25 indexing happens in-memory at first request.

Query Pipeline (runtime)

  1. Query — user submits question via chat UI
  2. Mode detection — regex patterns classify intent (answer/generate/rca/review)
  3. Hybrid search — BM25 sparse retrieval + dense vector search (Qdrant)
  4. RRF Fusion — Reciprocal Rank Fusion combines both result lists
  5. Rerank — top-6 chunks selected, filtered by optional source type
  6. LLM generation — chunks injected as numbered context, Groq streams the answer
  7. Cited answer — response with [n] citations, source badges shown in UI

Each chunk has the following TypeScript interface:

export interface KBChunk {
  id: string;                    // unique identifier e.g. "sel-001"
  text: string;                   // full text content for BM25 indexing
  source_type: SourceType;       // one of 9 types
  path: string;                   // original file path
  heading?: string;              // section heading
  line_start?: number;           // line number in source file
  ticket_key?: string;           // JIRA ticket e.g. "VWO-1234"
  tc_id?: string;               // test case ID e.g. "TC-VWO-0001"
  build_id?: string;            // CI build e.g. "BUILD-4521"
}

BM25 Retrieval

The BM25 (Best Match 25) implementation in lib/bm25.ts is a pure TypeScript in-memory search engine. It uses the classic Okapi BM25 formula: 

// BM25 scoring formula for term t in document d:
score(d, q) = Σ IDF(t) × (tf(t,d) × (k1 + 1)) / (tf(t,d) + k1 × (1 - b + b × |d|/avgdl))

// Parameters:
k1 = 1.5  // term frequency saturation
b  = 0.75 // length normalization (0=no norm, 1=full norm)

Indexing process

const { index } = getIndex(); // lazy-initialized singleton
// First call: iterates all KB chunks, tokenizes, builds inverted index
// Subsequent calls: returns cached index (sub-millisecond)

Search with source filtering

const chunks = search(query, topK=6, sourceFilter=["selenium", "playwright"]);
// Returns KBChunk[] sorted by BM25 score, filtered by source_type

The tokenizer lowercases text, removes punctuation, and filters tokens shorter than 3 characters. This produces clean term frequencies for the inverted index.

LLM & Prompt Engineering

The prompt system (lib/prompts.ts) auto-detects the query mode and builds mode-specific system prompts:

ModeTrigger PatternSystem Behavior
answer(default)Practical answer with code examples, bullet points
generategenerate|create|write|draftStructured test cases with all required fields
rcaroot cause|flaky|failure|debugCategorized RCA with evidence and fix suggestion
reviewreview|coverage|gap|missingCoverage analysis with requirement cross-refs

All modes enforce the same core rules:

  • Answer only from provided context chunks — no hallucination
  • Cite every claim with [n] referencing the chunk number
  • If not in context, say so explicitly

API Endpoints

POST /api/chat

Streaming chat endpoint. Accepts messages array and optional sourceFilter. Returns SSE stream via Vercel AI SDK data protocol.

// Request body
{
  messages: [{ role: "user", content: "How do I fix flaky tests?" }],
  sourceFilter: ["jenkins", "jira"]  // optional
}

// Response headers
X-Citations: '[{"n":1,"label":"BUILD-4499","source_type":"jenkins","path":"..."}]'
X-Mode: "rca"
X-Chunks-Used: "6"

GET /api/health

Health check endpoint. Returns KB status and API key availability.

{ status: "ok", kb_chunks: 25, sources: [...], groq_key_set: true, timestamp: "..." }

GET /api/stats

Statistics endpoint. Returns chunk counts by source type.

{ total_chunks: 25, by_source: { selenium: 5, playwright: 5, ... }, sources_active: 9 }

Vercel Deployment

The app is designed for zero-configuration Vercel deployment:

# 1. Clone repository
git clone https://github.com/SravanaKumarSanka/qabuddy-ai
cd qabuddy-ai

# 2. Install dependencies
npm install

# 3. Set environment variable
export GROQ_API_KEY=your_key_here

# 4. Run locally
npm run dev

# 5. Deploy to Vercel
npx vercel --prod

Environment Variables

VariableRequiredDescription

vercel env add GROQ_API_KEY production

Extending the Knowledge Base

To add new content, append chunks to the knowledgeBase array in lib/knowledge-base.ts:

{
  id: "mymodule-001",
  source_type: "docs",
  path: "data/my-doc.md",
  heading: "My Section",
  text: `[my-doc] Section heading

  Your content here. BM25 will index this automatically.
  Include relevant keywords, code snippets, and context.
  The more specific, the better the retrieval quality.`
}

The BM25 index rebuilds automatically on the next request after deployment. No vector database setup required for the Vercel version.

Original Design (AITesterBlueprint3x)

QABuddy.ai is adapted from chapter_08_QABuddyAI in the PramodDutta/AITesterBlueprint3x repository by Pramod Dutta.

The original system uses:

  • BGE-M3 embeddings — BAAI/bge-m3 model for dense + sparse vectors (ColBERT-style)
  • Qdrant — vector database running locally or on Qdrant Cloud
  • Flask backend — Python web server with Server-Sent Events streaming
  • Groq API — openai/gpt-oss-120b model for generation
  • Hybrid retrieval — RRF fusion of dense + sparse Qdrant search results

Cloud Adaptation Decisions

The following changes were made to enable Vercel serverless deployment:

OriginalVercel AdaptationReason
BGE-M3 embeddingsBM25 (TypeScript)No 4GB ML model on serverless
Qdrant vector DBIn-memory BM25 indexNo persistent storage on Edge
Flask (Python)Next.js 14 API routesNative Vercel framework
Dynamic ingestionStatic knowledge-base.tsPre-compiled for fast cold start
gpt-oss-120bllama-3.1-70b-versatileAvailable on Groq free tier
Custom SSEVercel AI SDK streamTextProduction-grade streaming

The core RAG architecture concept (hybrid retrieval → rerank → LLM with citations) is preserved. Production use cases with more than a few hundred chunks should use the full original stack with Qdrant + BGE-M3 embeddings.
